1.2 Explication pour le rapport

  "Le diagramme de cas d'utilisation de Sahel.ai présente trois acteurs
  principaux. Le Visiteur accède aux pages publiques et peut interagir avec
  les vitrines digitales et le chatbot. Le Propriétaire, authentifié via
  JWT, gère ses commerces, ses documents et consulte les analytics. Le
  Client interagit avec l'assistant IA via WhatsApp ou Telegram, sans
  passer par l'interface web."

1.3 Questions potentielles du jury

  Q : "Pourquoi avoir séparé Visiteur et Client ?"
  R : "Le Visiteur est un utilisateur web anonyme. Le Client est une
  personne qui interagit via messagerie (WhatsApp/Telegram). Ils ont des
  cas d'utilisation différents : le visiteur navigue sur le site, le
  client envoie des messages à l'assistant IA."

  Q : "Un Propriétaire peut-il aussi être Visiteur ?"
  R : "Oui, un propriétaire connecté peut aussi consulter les pages
  publiques. Mais dans le diagramme, on modélise les rôles, pas les
  personnes physiques. Un même utilisateur peut avoir plusieurs rôles."

3.2 Explication

  "Ce diagramme de séquence montre le traitement complet d'une requête IA.
  La question de l'utilisateur est d'abord analysée pour détecter sa langue.
  Ensuite, le pipeline RAG est déclenché : la question est convertie en
  vecteur (embedding), une recherche sémantique est effectuée dans FAISS
  pour trouver les 3 passages les plus pertinents, et un prompt enrichi
  de ce contexte est envoyé à Groq API (Llama 3). La réponse est streamée
  au client et sauvegardée dans l'historique."

4.2 Explication pour le rapport

  "Le diagramme de classes de Sahel.ai modélise six entités principales.
  La classe Owner représente le propriétaire du commerce avec ses méthodes
  d'authentification. Business est la classe centrale qui contient tous
  les attributs métier (nom, type, ville, horaires). Document stocke les
  fichiers uploadés et le nombre de chunks FAISS générés. Conversation et
  Message tracent toutes les interactions avec les clients. Inquiry gère
  les demandes de contact indépendantes."

4.3 Questions potentielles du jury

  Q : "Pourquoi avoir séparé Conversation et Message ?"
  R : "Une conversation regroupe plusieurs messages d'un même client.
  Cela permet de suivre l'historique des échanges et de calculer des
  métriques comme le nombre de conversations totales."

  Q : "Pourquoi Inquiry n'est lié à aucune table ?"
  R : "Une inquiry est une demande de contact d'un visiteur qui n'a pas
  forcément de compte propriétaire. C'est une entité indépendante
  volontairement."

  "Le MCD de Sahel.ai comporte six entités. La table Owner est en relation
  un-à-plusieurs avec Business (un propriétaire peut gérer plusieurs
  commerces). Business a deux sous-ensembles : Document (un commerce peut
  avoir plusieurs documents PDF) et Conversation (un commerce reçoit
  plusieurs conversations). Chaque Conversation contient un ou plusieurs
  Message. La table Inquiry est indépendante car elle correspond à des
  demandes de contact externes, sans lien avec un compte propriétaire."

5.3 Règles de gestion

  RG1 : Un propriétaire peut créer 0 ou plusieurs commerces.
  RG2 : Un commerce appartient à un et un seul propriétaire.
  RG3 : Un commerce peut avoir 0 ou plusieurs documents uploadés.
  RG4 : Un commerce peut avoir 0 ou plusieurs conversations.
  RG5 : Une conversation peut avoir 1 ou plusieurs messages.
  RG6 : Une inquiry (demande de contact) est indépendante d'un compte.
