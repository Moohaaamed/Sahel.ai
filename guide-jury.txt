================================================================================
1. PRÉSENTATION GÉNÉRALE DU PROJET
================================================================================

Sahel.ai est une plateforme SaaS qui permet aux petites et moyennes entreprises
(PME) marocaines de:
  - Créer un mini-site vitrine professionnel en 2 minutes
  - Intégrer un assistant IA multilingue (Français, Arabe/Darija, Anglais)
    qui répond automatiquement aux questions des clients
  - Gérer les documents (brochures, menus, tarifs) pour enrichir les réponses
  - Recevoir et gérer les demandes de devis/contact (inquiries)
  - Générer des QR codes pour partager le mini-site
  - Utiliser WhatsApp et Telegram comme canaux supplémentaires

Le projet utilise:
  - Backend: Python FastAPI avec RAG (Retrieval-Augmented Generation)
  - Frontend: React 19 + Vite
  - Base de données: PostgreSQL (Supabase) avec SQLAlchemy
  - LLM: Groq (llama-3.3-70b-versatile) pour les réponses IA
  - Embeddings: sentence-transformers (all-mpnet-base-v2)
  - Vector store: FAISS pour la recherche sémantique
  - Déploiement: Frontend sur Vercel, Backend sur HuggingFace Spaces

Pourquoi ce projet est pertinent pour le PFE:
  - Problème réel: les PME marocaines n'ont pas les moyens d'avoir un site web
    avec chatbot. Sahel.ai résout ce problème avec une solution clé en main.
  - Stack technique complet: frontend, backend, IA, base de données, déploiement
  - Multilingue: gère le Darija (arabe dialectal marocain) ce qui est innovant
  - Architecture moderne: RAG, embeddings, streaming, WebSockets


================================================================================
2. ARCHITECTURE TECHNIQUE
================================================================================

   [Utilisateur]
        |
   [Navigateur Web] / [WhatsApp] / [Telegram]
        |
   +----+-------------+-----+
   |    |             |     |
[Vercel]       [WhatsApp Bot]  [Telegram Bot]
(Frontend)     (Node.js)       (Node.js)
   |                |             |
   |           [Backend API] ------+
   |                |     (FastAPI - HuggingFace Spaces)
   |           [Base de données]
   |           (Supabase PostgreSQL)
   |
   +---> Groq Cloud (LLM llama-3.3-70b)
   +---> sentence-transformers (Embeddings)
   +---> FAISS (Vector Search)

FLUX D'UNE REQUÊTE CHAT (RAG):
  1. L'utilisateur pose une question dans le chat
  2. Le frontend envoie la question à /businesses/{id}/chat
  3. Le backend détecte la langue (ar/fr/en) via detect_language()
  4. Le backend cherche les documents pertinents dans FAISS (similarity_search)
  5. Le backend construit un prompt avec: la question + les documents trouvés +
     le profil complet du commerce (city, horaires, services, highlights...)
  6. Le backend envoie le prompt à Groq (llama-3.3-70b-versatile)
  7. Groq génère une réponse naturelle
  8. La réponse est sauvegardée dans la base de données
  9. Le frontend affiche la réponse

FLUX CRÉATION D'UN COMMERCE:
  1. L'utilisateur remplit le formulaire dans le dashboard
  2. Le backend reçoit POST /businesses avec les données
  3. Le backend normalise et sauvegarde dans PostgreSQL
  4. Le commerce apparaît immédiatement dans la liste
  5. L'utilisateur peut uploader des documents (PDF) pour enrichir le RAG
  6. Le mini-site est accessible via /b/{slug}

3.1 backend/backend_chatdoc.py (1993 lignes)
--------------------------------------------------
C'est le fichier principal du projet. Il contient TOUTE l'API REST.

CE QU'IL FAUT RETENIR:

a) Les modèles Pydantic (lignes 109-216):
   Ce sont des "contrats" qui définissent la structure des données entrantes
   et sortantes. FastAPI les utilise automatiquement pour:
   - Valider les données (vérifier qu'un email est bien un email)
   - Générer la documentation Swagger (accessible via /docs)
   - Convertir les types (string en int, etc.)
   
   BusinessCreate (L109): c'est ce que l'utilisateur envoie quand il crée
   un commerce. On y trouve: name, description, city, working_hours, etc.
   
   Business (L133): c'est ce que l'API renvoie. Il contient tout ce qu'il y a
   dans BusinessCreate + des champs auto-générés comme id, slug, site_url.

b) Détection de langue — detect_language() (L635):
   C'est une fonction cruciale pour le multilingue. Elle fonctionne en 2 étapes:
   1. Si le texte contient des caractères arabes (\u0600-\u06FF), retourne "ar"
   2. Sinon, elle compare avec des dictionnaires de mots-clés pour le français
      et l'anglais, et un ensemble de mots Darija (dialecte marocain) écrits
      en lettres latines comme "had", "dyal", "chnu", "ach", "bzf", "mzyan"
   
   IMPORTANT: La détection du Darija a été améliorée car les phrases comme
   "had l business dyalach" étaient détectées comme anglais à cause des mots
   "business" et "had". On a ajouté ~40 mots-clés Darija pour corriger ça.

c) RAG — answer_for_business() (L1068):
   C'est le coeur de l'assistant IA. Voici ce qu'elle fait:
   1. Charge le vector store FAISS pour ce commerce (via load_store)
   2. Cherche les 3 documents les plus pertinents (similarity_search avec k=3)
   3. Construit un contexte avec les documents trouvés
   4. Ajoute les infos du profil (city, working_hours, primary_services, etc.)
   5. Ajoute une instruction de langue (répondre dans la même langue que
      la question)
   6. Envoie le tout à Groq
   7. Sauvegarde la question+ réponse dans la base
   8. Retourne la réponse

d) Streaming — stream_answer_for_business() (L1126):
   Version améliorée qui envoie la réponse caractère par caractère via
   WebSocket ou Server-Sent Events. Cela permet d'afficher la réponse
   en temps réel sans attendre qu'elle soit complète. L'utilisateur voit
   les tokens arriver un par un.

e) Analytics — analytics_for_business() (L1240):
   Calcule les statistiques pour un commerce:
   - Nombre total de messages
   - Nombre total de conversations
   - Langues utilisées (ar/fr/en)
   - Questions les plus fréquentes (top 5)
   - 8 derniers messages

   ET analytics pour tout l'owner — /owner/analytics (L1532):
   Regroupe les stats de TOUS les commerces en une seule requête.
   C'est plus rapide que de faire 13 appels séparés.

f) Authentification:
   - create_owner_session() (L354): crée un JWT token avec l'ID du propriétaire
   - verify_owner_token() (L368): vérifie le JWT et retourne le propriétaire
   - require_owner() (L382): utilisé comme middleware sur les endpoints protégés
   
   Le JWT est stocké dans localStorage côté frontend et envoyé dans le header
   Authorization: Bearer <token>. Il expire après 24h.

g) Processus de création d'un commerce (POST /businesses, L1429):
   1. Vérifie que le slug est unique (ex: "amoudou", "saghro")
   2. Crée les URLs (site_url = "/b/saghro", chat_url = "/chat/saghro")
   3. Sauvegarde dans la base

h) Upload de documents (POST /businesses/{id}/documents, L1697):
   1. Reçoit le fichier PDF/TXT
   2. Extrait le texte avec pypdf
   3. Découpe en chunks de 1000 caractères (avec 100 de chevauchement)
   4. Crée les embeddings avec all-mpnet-base-v2
   5. Indexe dans FAISS
   6. Sauvegarde les métadonnées (nom, taille, nombre de chunks)

i) Mini-site dynamique (GET /b/{slug}, L1958):
   Utilise Jinja2 pour générer une page HTML complète avec:
   - Les informations du commerce
   - Le chat intégré
   - Les traductions (français/arabe)
   - Le widget embed si configuré

j) Widget embed (GET /embed/{slug}.js, L1873):
   Génère un script JavaScript dynamique que les commerces peuvent ajouter
   à leur site existant. Ce script injecte un bouton de chat flottant.

LES QUESTIONS PIÈGES POSSIBLES:

Q: "Comment fonctionne la recherche de documents?"
R: "On utilise un vector store FAISS. Chaque document est découpé en chunks
    de 1000 caractères. Chaque chunk est converti en vecteur numérique
    (embedding) de 768 dimensions via le modèle all-mpnet-base-v2.
    Quand un client pose une question, on convertit aussi sa question en
    vecteur, et FAISS trouve les 3 chunks les plus proches dans l'espace
    vectoriel. Ces chunks sont donnés comme contexte au LLM."

Q: "Pourquoi FAISS et pas une autre solution?"
R: "FAISS est léger, open-source (Facebook/Meta), et fonctionne très bien
    en local. Pas besoin de service externe comme Pinecone. Pour notre
    volume de données (quelques centaines de documents), c'est parfait.
    Le seul inconvénient est qu'il faut rebuild l'index quand on ajoute
    un document, mais ça prend moins d'une seconde."

Q: "Comment gérez-vous le multilingue?"
R: "On détecte la langue de la question avec detect_language() qui utilise
    à la fois la détection de caractères arabes et un scoring par mots-clés
    pour le français, l'anglais et le darija. Ensuite on passe une instruction
    de langue au LLM via le système prompt pour qu'il réponde dans la même
    langue que la question. Pour le darija, on a enrichi le prompt avec des
    instructions spécifiques."

Q: "Pourquoi avez-vous utilisé un Jeton JWT plutôt que des sessions classiques?"
R: "Le JWT est stateless: le serveur n'a pas besoin de stocker les sessions.
    C'est plus scalable. Le token contient l'ID du propriétaire et expire
    après 24h. C'est signé avec une clé secrète (APP_SECRET_KEY)."

3.2 backend/storage.py (240 lignes)
--------------------------------------------------
Couche d'abstraction entre l'API et la base de données.

CE QU'IL FAUT RETENIR:

a) Modèles SQLAlchemy (L33-120):
   Ce sont les "tables" traduites en code Python:
   - OwnerModel: id, full_name, email, password_hash, verified...
   - BusinessModel: id, owner_id, slug, name, description, city...
   - DocumentModel: id, business_id, file_name, file_type...
   - ConversationModel: id, business_id, language, started_at...
   - MessageModel: id, business_id, conversation_id, question, answer...
   - InquiryModel: id, business_id, name, contact, message, status...

b) init_database() (L174):
   Crée automatiquement les tables au démarrage du serveur si elles
   n'existent pas encore (grâce à AUTO_CREATE_TABLES=true). C'est ce qui
   permet de déployer sans avoir à lancer de migrations à la main.

c) run_migrations() (L135):
   Ajoute les colonnes manquantes aux tables existantes. Utile quand on
   ajoute des champs (ex: city, highlights) après la création initiale.

d) read_table / write_table (L205-223):
   Fonctions génériques qui marchent avec TOUTES les tables.
   - read_table: SELECT * FROM table
   - write_table: DELETE FROM table + INSERT INTO table (remplacement complet)

e) import_json_table (L232):
   Au démarrage, si une table est vide et que le fichier JSON correspondant
   existe, il importe les données. C'est comme ça que les données de seed
   sont chargées (businesses.json, messages.json, etc.)

ASTUCES À CONNAÎTRE:
- Le fichier utilise PostgreSQL via l'URL DATABASE_URL. Si l'URL contient
  "postgres://", normalize_database_url() (L15) le remplace automatiquement
  en "postgresql+psycopg2://" qui est le format attendu par SQLAlchemy.
- La variable SessionLocal est un "sessionmaker" qui crée des sessions
  de base de données. Chaque requête ouvre une session, fait ses opérations,
  et la ferme.

4.1 website/src/components/DashboardPage.jsx (~1400 lignes)
--------------------------------------------------
C'est LE fichier le plus important pour la présentation car c'est
l'interface que l'utilisateur voit après connexion.

STRUCTURE:
a) États (useState, L87-134):
   Chaque "useState" crée une variable qui, quand elle change, met à jour
   l'affichage automatiquement. Les principales:
   - businesses: liste des commerces (tableau)
   - businessStats: stats par commerce (objet)
   - recentInteractions: dernières conversations
   - topQuestions: questions fréquentes
   - languageDistribution: répartition des langues
   - editBusiness: commerce en cours d'édition

b) Hooks de cycle de vie (useEffect):
   - Un useEffect au chargement: vérifie la session, charge les commerces
   - Un useEffect: charge les analytics dès que les commerces sont prêts

c) Fonctions principales:
   - loadDashboardData() (L159): appel unique à /owner/analytics qui
     retourne TOUTES les stats en une seule requête (messages, questions,
     langues, interactions récentes)
   - createBusiness() (L300): appel POST pour créer un commerce
   - updateBusiness() (L355): appel PUT pour modifier un commerce
   - copyLink() / downloadQrCode(): partage / QR code

d) Affichage:
   - 4 cartes de stats en haut (Conversations, Taux de réponse,
     Commerces actifs, Langues)
   - Tableau des commerces avec recherche
   - Graphiques: questions fréquentes, répartition des langues
   - Liste des interactions récentes

e) Composant AnimatedNumber (L62):
   Fait défiler les chiffres de 0 jusqu'à la valeur réelle (effet d'animation)
   Bug important qu'on a corrigé: au début il ne s'animait qu'une fois et
   restait bloqué à 0 quand les données arrivaient après le premier rendu.

EXPLICATION DU FLUX DE DONNÉES:
1. L'utilisateur arrive sur le dashboard
2. businesses = [] (vide) et loading = true (skeleton affiché)
3. Le useEffect lance:
   a) GET /businesses?owner_id=... → charge la liste des commerces
   b) GET /owner/analytics → charge les stats
4. Quand les stats arrivent → AnimatedNumber s'anime de 0 vers la valeur
5. Quand les commerces arrivent → tableau affiché

LES QUESTIONS PIÈGES:
Q: "Pourquoi les stats mettaient du temps à s'afficher?"
R: "Au début, loadDashboardData faisait une requête par commerce (13 appels).
   On a créé un endpoint /owner/analytics qui regroupe tout en un seul appel.
   Le temps est passé de ~10s à <1s."

Q: "Pourquoi les chiffres restaient bloqués à 0?"
R: "Le composant AnimatedNumber utilisait une variable started qui empêchait
   la ré-animation. Quand la valeur passait de 0 à 22, le composant ignorait
   le changement car started était déjà true. On a remplacé par un système
   qui compare l'ancienne et la nouvelle valeur."

--------------------------------------------------
Composant de chat réutilisable. Utilisé à la fois dans:
- Le dashboard (pour tester le chat)
- Les mini-sites publics

Fonctionnement:
1. L'utilisateur tape un message
2. sendMessage() (L169): POST /businesses/{id}/chat avec {question, conversation_id}

================================================================================
5. BASE DE DONNÉES
================================================================================

5.1 Choix de PostgreSQL
--------------------------------------------------
On utilise PostgreSQL via Supabase (service cloud gratuit). Le choix est
justifié par:
- Professionnel: PostgreSQL est la base de données la plus utilisée en
  production. Supabase offre 500MB de données gratuitement.
- Migration depuis SQLite: Au début du projet, on utilisait SQLite (fichier
  local) pour le développement. On a migré vers PostgreSQL pour le
  déploiement production.
- ORM: SQLAlchemy permet de changer de base sans changer le code.

5.2 Structure des données
--------------------------------------------------
6 tables principales:

1. owners - Les propriétaires de commerces
   id (PK), full_name, email (UNIQUE), password_hash, verified, etc.

2. businesses - Les commerces
   id (PK), owner_id (FK→owners), slug (UNIQUE), name, description,
   city, working_hours, primary_services, highlights, public_knowledge,
   address, latitude, longitude, cover_image_url, etc.

3. documents - Les fichiers uploadés
   id (PK), business_id (FK→businesses), file_name, file_type,
   file_url, chunks (nombre de chunks), size_bytes

4. conversations - Les sessions de chat
   id (PK), business_id (FK→businesses), language (détectée),
   visitor_ip, started_at

5. messages - Les messages individuels
   id (PK), business_id (FK→businesses), conversation_id (FK→conversations),
   question, answer, sources (JSON), language, created_at

6. inquiries - Les demandes de contact
   id (PK), business_id (FK→businesses), name, contact, message,
   status (new/contacted/closed), created_at

5.3 Comment les données de seed arrivent sur un nouveau déploiement
--------------------------------------------------
1. Les fichiers JSON (businesses.json, documents.json, messages.json,
   conversations.json, owners.json, inquiries.json) sont commités dans git
2. Dans Dockerfile: COPY backend/data/ /data/data/ copie ces fichiers
3. Au démarrage, init_database() crée les tables
4. import_json_table() vérifie: si table vide ET fichier JSON existe →
   importe les données
5. Les commerces, documents et messages de démo sont chargés
6. Le LLM peut répondre sur ces commerces car leur profil complet est
   dans la base de données

6. DÉPLOIEMENT
================================================================================

6.1 Architecture de déploiement
--------------------------------------------------
```
  Frontend (React) → Vercel (gratuit)
  Backend (FastAPI) → HuggingFace Spaces (gratuit - Docker)
  Base de données → Supabase (gratuit - 500MB)
  LLM → Groq Cloud (gratuit - 30 req/min)
```

6.2 Pourquoi ce choix?
--------------------------------------------------
- Vercel + HF Spaces + Supabase: tout est gratuit, pas besoin de carte
  bancaire. Parfait pour un PFE.
- Frontend et backend séparés: scaling indépendant. Si le frontend a du
  traffic, ça n'affecte pas le backend.
- Docker pour le backend: HF Spaces nécessite un Dockerfile. Il contient
  Python 3.12, les dépendances, et le code.

6.3 Backend sur HuggingFace Spaces
--------------------------------------------------
Le Dockerfile (à la racine) fait:
1. python:3.12-slim comme base (image légère)
2. Installe build-essential (nécessaire pour faiss et sentence-transformers)
3. Installe les dépendances Python via requirements.txt
4. Copie le dossier backend/ dans /app/
5. Copie les données seed dans /data/data/ (pour import au démarrage)
6. Configure DATA_ROOT=/data, AUTO_CREATE_TABLES=true
7. Lance uvicorn sur le port 7860 (port attendu par HF Spaces)

Variables d'environnement sur HF Spaces (à configurer dans les secrets):
- GROQ_API_KEY: clé pour le LLM
- APP_SECRET_KEY: clé pour les JWT
- DATABASE_URL: URL Supabase PostgreSQL
- CORS_ORIGINS: domaines autorisés
- SMTP_*: pour les emails de vérification

Mise à jour via git:
   git remote add hf https://huggingface.co/spaces/username/sahel-api
   git push hf main --force

Problème rencontré: HF Spaces rejette les fichiers binaires (images, gifs).
On a dû créer une branche propre sans les fichiers binaires de l'historique
git pour pouvoir déployer.

6.4 Frontend sur Vercel
--------------------------------------------------
Connecté au repo GitHub. Chaque push sur main déclenche un déploiement.

Variable d'environnement sur Vercel:
- VITE_API_URL: URL du backend sur HF Spaces (ex: https://sahel-api.hf.space)

SPA Routing: Vercel redirige toutes les URLs vers index.html grâce aux
rewrites dans vercel.json. Le HashRouter (#/) gère le reste côté client.

6.5 Base de données Supabase
--------------------------------------------------
Service cloud PostgreSQL avec:
- Interface web pour les données
- API REST optionnelle
- 500MB de données (gratuit)
- Connexion via DATABASE_URL

Configuration: DATABASE_URL=postgres://user:pass@host:5432/postgres
Le backend utilise psycopg2 pour se connecter avec SSL.

6.6 Processus complet de déploiement
--------------------------------------------------
Quand on pousse sur main:
  1. GitHub reçoit le code
  2. Vercel détecte le changement et rebuild le frontend
     - npm install + npm run build
     - Sert le dossier dist/
  3. HF Spaces détecte le changement et rebuild le backend
     - Docker build avec le Dockerfile
     - Lance le conteneur avec les secrets
     - Au démarrage: init_database() + import_json_table()
     - L'API est disponible sur https://sahel-api.hf.space

7. QUESTIONS FRÉQUENTES DU JURY — RÉPONSES PRÉPARÉES
================================================================================

Q1: "Qu'est-ce qui différencie Sahel.ai des autres solutions?"
R: "Trois points: 1) C'est 100% marocain avec support du Darija. 2) C'est
    clé en main: mini-site + chatbot IA + QR codes + WhatsApp + Telegram
    en une seule plateforme. 3) Le RAG permet au chatbot de répondre avec
    les documents du commerce lui-même, pas des réponses génériques."

Q2: "Comment le chatbot IA fonctionne-t-il exactement?"
R: "Quand un client pose une question, on détecte d'abord la langue. Ensuite,
    on cherche dans les documents du commerce (via FAISS) les passages les
    plus pertinents. On construit un prompt qui contient ces passages + le
    profil du commerce. On envoie ça à Groq (llama-3.3-70b) qui génère une
    réponse naturelle et précise. La réponse est sauvegardée dans l'historique."

Q3: "C'est quoi le RAG?"
R: "Retrieval-Augmented Generation. Au lieu de juste demander au LLM de
    répondre de mémoire (ce qui peut donner des réponses fausses), on va
    d'abord chercher l'information dans les documents du commerce. Le LLM
    génère sa réponse à partir de ces documents. Ça garantit que les
    réponses sont factuelles et basées sur les vrais documents du commerce."

Q4: "Pourquoi avoir choisi FastAPI plutôt que Django?"
R: "FastAPI est plus léger, plus rapide, et supporte nativement l'async
    (asynchrone). C'est important pour le streaming des réponses du LLM.
    Django serait trop lourd pour une API REST pure. FastAPI génère aussi
    automatiquement la documentation Swagger (/docs)."

Q5: "Comment gérez-vous les performances?"
R: "Les embeddings sont faits en local avec sentence-transformers. La
    recherche vectorielle utilise FAISS qui est très rapide (<10ms). Le
    LLM est externalisé chez Groq car il nécessite un GPU puissant. Le
    frontend est statique sur Vercel (CDN mondial)."

Q6: "Comment vous assurez-vous que le chatbot ne raconte pas n'importe quoi?"
R: "On utilise le RAG: le LLM ne répond qu'à partir des documents qu'on
    lui fournit. Le prompt système lui demande explicitement de ne pas
    inventer d'informations. Si aucun document pertinent n'est trouvé, il
    répond qu'il n'a pas l'information et propose de contacter le commerce."

Q7: "Qu'est-ce qui se passe si Groq est indisponible?"
R: "On a une fallback. Le système peut fonctionner avec un modèle local
    via le dossier frontend-chat/. En production, on pourrait aussi
    basculer sur OpenAI ou un autre fournisseur. L'abstraction est déjà
    faite dans le code."

Q8: "Comment les données sont-elles sécurisées?"
R: "Les mots de passe sont hashés avec PBKDF2 (pas stockés en clair).
    L'authentification utilise des JWT signés avec une clé secrète. Les
    endpoints sont protégés par require_owner() qui vérifie le token.
    Les CORS sont configurés pour n'autoriser que des domaines spécifiques."

Q9: "Comment avez-vous géré le multilingue, spécialement le Darija?"
R: "Le Darija marocain est un défi car il n'a pas de standard écrit. On
    a une fonction detect_language() qui utilise des dictionnaires de
    mots-clés (had, dyal, chnu, bzf, wah, mzyan, etc.). Si le texte
    contient ces mots, on le traite comme du Darija. On a aussi enrichi
    le prompt système avec des instructions en arabe pour guider le LLM."

Q10: "Comment le mini-site est-il généré?"
R: "Il y a deux versions. Une version server-side avec Jinja2 (GET /b/{slug})
    qui génère le HTML côté backend avec les données du commerce. Et une
    version client-side avec PublicBusinessPage.jsx pour ceux qui visitent
    via le frontend React. Les deux sont synchronisées."

Q11: "Expliquez le flux d'upload d'un document"
R: "L'utilisateur upload un PDF. Le backend extrait le texte avec pypdf,
    le découpe en chunks de 1000 caractères (avec 100 de chevauchement pour
    garder le contexte). Chaque chunk est converti en vecteur avec
    sentence-transformers (all-mpnet-base-v2) qui produit 768 dimensions.
    Les vecteurs sont indexés dans FAISS. Quand un client pose une question,
    on cherche les 3 chunks les plus similaires."

Q12: "Comment avez-vous géré les problèmes de déploiement?"
R: "Le plus gros problème était que HF Spaces rejette les fichiers binaires
    dans git. On a dû créer une branche propre sans les images du commit
    initial. Aussi, le Dockerfile copiait les seed data dans /app/ mais
    DATA_ROOT pointait vers /data/. On a ajouté une ligne COPY pour copier
    les fichiers au bon endroit."

Q13: "Quels sont les limites et perspectives d'amélioration?"
R: "Limites: Le nombre de requêtes Groq est limité (30/min gratuit). FAISS
    doit être rebuildé à chaque ajout de document. Pas de paiement intégré.
    Perspectives: Multi-bot (un chatbot par commerce), authentification
    avancée, paiement en ligne, statistiques avancées, application mobile."

Q14: "Pourquoi avoir séparé le frontend et le backend?"
R: "Architecture microservices: le frontend React peut être déployé sur
    Vercel (CDN), le backend sur HF Spaces. Si un crash, l'autre continue.
    On peut mettre à jour l'un sans casser l'autre. Et le backend peut
    servir à la fois le site web, WhatsApp et Telegram."

Q15: "Le projet utilise Groq, c'est quoi?"
R: "Groq est un fournisseur de LLM qui a des puces spécialisées (LPU)
    pour l'inférence. Leur modèle llama-3.3-70b est très rapide et
    gratuit pour les petits volumes. On peut envoyer jusqu'à 30 requêtes
    par minute sans payer."
